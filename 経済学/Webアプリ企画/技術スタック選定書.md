# 配属マッチングWebアプリ 技術スタック選定書（Python 3.13 / Flask版）

- 作成日: 2026-07-11（改訂2: Python 3.13固定・画面と処理を単一フレームワークで実装する方針に変更）
- 前提条件
  - **Python 3.13 単一言語で完結し、管理コストを最小化する**
  - **画面（テンプレート）と処理（アルゴリズム）を1つのフレームワークで一括実装する**
  - 既存Python実装（DAアルゴリズム）をそのまま活用
  - 当面は認証なしで一般公開（データ永続化なし＝DB不要）
  - ランニングコストは無料枠内
  - ソース管理はGitHub
- アプリ構成: 希望順位を入力 → マッチング結果を出力。オプションでDAの過程（提案→保留→棄却）を可視化

---

## 1. 推奨構成（結論）

| レイヤ | 採用技術 | 理由 |
|---|---|---|
| 言語 | **Python 3.13** | 全レイヤ統一 |
| Webフレームワーク | **Flask 3.1**（Python 3.9+対応） | 画面（Jinja2テンプレート）と処理を1フレームワークで一括実装。最も枯れたリクエスト/レスポンスモデルで管理コスト最小 |
| テンプレート | **Jinja2**（Flask同梱） | フォーム・結果表・SVG図をサーバーサイドで描画 |
| フォーム処理 | **WTForms（Flask-WTF）** | 選好順位入力のバリデーションをPythonで宣言的に記述 |
| 可視化 | **サーバーレンダリングSVG ＋ ステップ送りURL**（後述） | JS不要でDA過程の再生を実現 |
| WSGIサーバ | **gunicorn** | Flask標準の本番構成 |
| ホスティング | **Render Free** | WSGIアプリと相性良好。GitHub連携自動デプロイ。無料 |
| CI/CD | **GitHub Actions**（pytest / ruff / mypy） | Pythonのみで完結 |
| パッケージ管理 | **uv** | pyproject.toml＋lockfileで再現性 |
| テスト | **pytest ＋ Hypothesis** | 安定性（ブロッキングペア0）・耐戦略性のプロパティテスト |

**リポジトリ構成**

```
haizoku-matching/
├── pyproject.toml
├── Dockerfile               # Cloud Run等への移行に備え最初から用意
├── app/
│   ├── __init__.py          # Flaskアプリファクトリ
│   ├── routes.py            # 入力 / 実行 / 結果 / 過程表示
│   ├── algorithms/          # 既存DA実装（ステップログ出力を追加）
│   ├── forms.py             # WTFormsによる選好入力定義
│   └── templates/           # Jinja2（入力・結果・過程再生・SVG部品）
├── tests/
└── .github/workflows/ci.yml
```

## 2. 過程可視化のJS不要設計（本構成の肝）

1. DA実行時に全ラウンドのイベントログ（誰が誰に提案→保留/棄却）をリストとして生成
2. 結果画面から「過程を見る」→ `/process/<run_id>?step=3` のようにステップ番号をURLで受け、そのステップ時点の状態をJinja2でSVG描画
3. 「前へ/次へ」は単なるリンク（`?step=2` / `?step=4`）。自動再生が欲しければ `<meta http-equiv="refresh">` で数秒ごとに次ステップへ遷移

- run_id はDBなしでも、選好データをURLセーフに符号化（または署名付きセッション/一時ファイル）すれば保持可能。MVPはFlaskの `session`（Cookie）で十分
- 将来リッチにしたければ、この設計のまま HTMX を1行追加するだけで部分更新化できる（テンプレートは流用）

## 3. フレームワーク比較（画面＋処理の一括実装が前提）

| 候補 | 評価 | 所見 |
|---|---|---|
| **Flask 3.1 + Jinja2** | ◎ | 軽量・最大級の情報量・自由な構成。DBなしMVPに過不足なし。「フォーム提出→計算→結果表示」はFlaskの教科書的ユースケース |
| Django 6.0 | ○ | 画面＋処理＋DB＋認証＋管理画面まで一括。ただしDBなしのMVPには重装備で、設定・規約の学習が先行コストになる。**Phase 3（認証＋永続化）で最有力の移行先** |
| FastAPI + Jinja2 | ○ | 技術的には可能だがFastAPIはAPI特化で、フォーム・セッション周りはFlaskの方が素直 |
| NiceGUI（前版の推奨） | △→変更 | 純PythonでリッチUIが作れるが、WebSocket常時接続・独自の状態同期モデルが管理対象を増やす。「枯れた一括実装」という今回の優先軸ではFlaskが優位 |
| Streamlit | △ | 全再実行モデルがステップ再生と相性が悪い（前版の評価どおり） |

**Flask vs Django の判断基準**: 現時点でDB・認証・管理画面が不要 → Flask。Phase 3でこれらが必要になった時点で、(a) Flaskに Flask-Login＋SQLAlchemy を足す、(b) Djangoに載せ替える、を再評価。テンプレート（Jinja2/DTL）は概念がほぼ同じため移行コストは限定的。

## 4. ホスティング（無料枠・WSGI前提で再確認）

| 候補 | 無料枠 | 所見 | 評価 |
|---|---|---|---|
| **Render Free** | 512MB / 月750時間 | gunicorn＋Flaskは最も標準的な構成。15分無トラフィックでスピンダウン（復帰約1分） | ◎ |
| Google Cloud Run | 常時無料枠 | WebSocketを使わないFlaskならゼロスケールと好相性（NiceGUI時代の懸念が消える）。Dockerfile運用 | ○（移行先の第一候補に昇格） |
| Fly.io | 無料相当枠 | 可だが設定コストで劣後 | ○ |
| PythonAnywhere | 無料プランあり | WSGI専用ホスティングでFlask対応（NiceGUIでは不可だった）。性能・制約面でRenderに劣後 | △ |

WebSocketを使わない構成にしたことで、ホスティングの選択肢と移行自由度がNiceGUI案より広がる点も副次的メリット。

## 5. トレードオフ（明示しておくこと）

- **得るもの**: 最小の依存・最大の情報量・枯れた運用モデル、ホスティング自由度、既存DA実装との完全な地続き
- **失うもの**: ページ遷移ベースのUI（画面内での動的な並べ替えやスムーズなアニメーションはなし）。順位入力はプルダウン列 or 順位番号入力で実現
- UI要件が高度化したら: ①HTMX追加（テンプレート流用・小コスト）→ ②フロント分離（React等）の順にエスカレーション可能。今の設計はどちらにも進める
